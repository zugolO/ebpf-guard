# Глава 24. Глоссарий терминов

> Уровень: **справочный**. Сводный глоссарий по всему пособию — объединяет
> и дополняет мини-глоссарии глав 1–23. У каждого термина указана глава,
> где он разбирается подробнее.

Каждый термин ниже — из реального словаря, который встречается в коде,
конфигурации или документации ebpf-guard, а не абстрактное определение
«из учебника». Список отсортирован по алфавиту (латиница, затем кириллица).

## A–C

- **ABI (Application Binary Interface)** — здесь: контракт из конкретного набора экспортируемых функций (`malloc`/`evaluate`/…) с фиксированными сигнатурами, которому обязан следовать любой `.wasm`-плагин. (Глава 16)
- **`action: drop`** — событие отбрасывается без генерации алерта; способ подавить шум на уровне самого правила. (Глава 8)
- **Adaptive sampling** — автоматическое увеличение семплирования при росте нагрузки на CPU. (Глава 6)
- **Anomaly score** — число `[0, 1]`, отражающее, насколько событие статистически не похоже на базовый профиль; сравнивается с `anomaly_threshold`. (Глава 9)
- **Anonymous executable memory region** — область памяти процесса с правом на исполнение, не отображённая ни на один файл на диске — типичный признак fileless-малвари. (Глава 23)
- **AppArmor-профиль** — LSM-политика, ограничивающая доступ процесса к файлам, сети и capabilities по имени пути/протокола. (Глава 20)
- **Attestation (аттестация BPF-программы)** — проверка, что загруженная в ядро программа имеет тот же криптографический тег (`bpf_prog_info.tag`), что и ожидаемая, то есть не была подменена. (Глава 15)
- **Baseline (базовый профиль)** — накопленное за период обучения представление «нормального» поведения конкретного workload'а; профайлер (глава 9) считает EWMA-статистику, а дрейф-детектор (глава 11) фиксирует пять множеств поведения после `baseline_window` — это два разных, но одноимённых понятия. (Главы 9, 11)
- **Bearer token** — токен авторизации HTTP API; при отсутствии в конфиге генерируется автоматически один раз при старте и печатается в лог. (Глава 3)
- **Best-effort conversion** — стратегия конвертации Falco-правил, при которой поддерживаемая часть условия переносится, а неподдерживаемая явно помечается как отброшенная, вместо отказа от всего правила целиком. (Глава 21)
- **`bpf2go`** — генератор типизированных Go-обёрток вокруг скомпилированного eBPF-байткода, запускается через `make generate`. (Главы 2, 3, 5)
- **BPF map** — структура ключ-значение, общая между eBPF-программой и userspace-кодом. (Глава 2)
- **BPF task iterator** — механизм ядра (`bpf iter/task`), позволяющий BPF-программе перечислить все задачи (`task_struct`) напрямую, без похода через `/proc`. (Глава 15)
- **BTF (BPF Type Format)** — метаданные о типах структур ядра, используемые для разрешения смещений полей. (Глава 2)
- **Bulk API** — пакетный протокол Elasticsearch/OpenSearch для вставки множества документов одним HTTP-запросом вместо одного запроса на документ. (Глава 14)
- **Canary file (файл-приманка)** — файл, специально размещённый там, где его никогда не открывают легитимные процессы, но часто открывают инструменты пост-эксплуатации. (Глава 15)
- **CAP_BPF / CAP_SYS_ADMIN / CAP_NET_ADMIN / CAP_PERFMON / CAP_IPC_LOCK** — Linux capabilities, дающие точечные привилегии вместо полного root: загрузка BPF-программ, доступ к BTF, сетевой enforcement, perf-события и блокировка памяти соответственно. (Глава 23)
- **Кардинальность метрики** — число уникальных комбинаций значений меток одной метрики; чрезмерная кардинальность перегружает Prometheus. (Глава 13)
- **`class: drift`** — метка YAML-правила (глава 8), означающая, что оно проходит через `profiler.DriftBaselineProfiler` и подавляется в период обучения — не путать с `internal/drift.Detector`. (Главы 8, 11)
- **ClientHello** — первое сообщение TLS-хендшейка, отправляемое клиентом: версия протокола, cipher suites, extensions — используется для JA3/JA4. (Глава 17)
- **CNI (Container Network Interface)** — плагин, отвечающий за сеть подов в Kubernetes; Cilium — один из примеров CNI с собственными eBPF-возможностями. (Глава 1)
- **Cobra** — Go-библиотека для построения CLI с подкомандами, флагами и авто-генерацией help; `ebpf-guard` целиком построен на ней. (Глава 18)
- **CompiledModule vs Instance** — в wazero компиляция байткода в машинный код (дорогая, один раз) и создание изолированного инстанса с собственной памятью (дешевле, на каждый вызов) — разделённые операции. (Глава 16)
- **Constant-time comparison** — сравнение строк за время, не зависящее от того, на каком символе найдено первое различие; защищает от восстановления секрета по замерам задержки ответа. (Глава 23)
- **CO-RE (Compile Once – Run Everywhere)** — техника, позволяющая одному скомпилированному eBPF-байткоду работать на разных версиях ядра благодаря относительным, разрешаемым во время загрузки обращениям к полям. (Глава 2)
- **`CorrelationEngine`** — центральный объект `internal/correlator/`, объединяющий `RuleEngine`, `Profiler` и `ShardedEventBuffer`. (Глава 4)
- **CORS preflight** — предварительный `OPTIONS`-запрос, который браузер шлёт перед «небезопасным» кросс-origin запросом, чтобы узнать, разрешён ли он сервером. (Глава 13)
- **Cosine distance (косинусное расстояние)** — мера непохожести направления двух векторов частот; `0` — идентичны, `1` — ортогональны. Используется `SequenceProfiler`-ом для сравнения векторов частоты syscall'ов. (Глава 9)
- **Coverage matrix** — таблица соответствия «техника MITRE ATT&CK → файл(ы) правил», `docs/coverage-matrix.md`. (Глава 8)

## D–I

- **DaemonSet** — объект Kubernetes, гарантирующий запуск одной копии пода на каждой (подходящей) ноде кластера; так ebpf-guard разворачивается в проде. (Главы 1, 20)
- **Deny-unknown (allowlist mode)** — модель детекции, где разрешено только явно замеченное во время обучения, а не «то, что не похоже на норму». (Глава 11)
- **DGA (Domain Generation Algorithm)** — техника вредоносного ПО/C2-инфраструктуры, генерирующая псевдослучайные доменные имена вместо жёстко закодированного адреса; распознаётся по высокой энтропии символов. (Главы 7, 17)
- **DNS tunneling** — техника эксфильтрации данных или C2-коммуникации через кодирование полезной нагрузки в поддоменах DNS-запросов. (Глава 17)
- **`--dry-run`** — режим запуска, в котором реальные eBPF-коллекторы заменяются на `SyntheticCollector`, генерирующий синтетические события без ядра и без root. (Глава 3)
- **eBPF (extended Berkeley Packet Filter)** — технология ядра Linux, позволяющая безопасно выполнять пользовательский байткод внутри ядра без написания модуля ядра. (Глава 1, подробно — глава 2)
- **Enforcer** — компонент, физически выполняющий действия (`block`/`kill`/`throttle`), запрошенные правилами через `action`; правило только просит, enforcer решает и делает. (Глава 12)
- **`eventCh`** — единственный Go-канал (`chan types.Event`), через который все коллекторы передают события в основной цикл; точка развязки между источником событий и логикой обработки. (Глава 4)
- **EWMA (Exponentially Weighted Moving Average)** — экспоненциально взвешенное скользящее среднее; даёт больший вес недавним наблюдениям. (Глава 9)
- **`exceptions`** — список условий-исключений в правиле: совпадение с любым из них подавляет алерт даже при совпадении основного условия. (Глава 8)
- **Falco condition** — булево выражение над полями события (`evt.*`, `fd.*`, `proc.*`, ...), определяющее, когда срабатывает Falco-правило. (Глава 21)
- **Fanout** — рассылка одного события сразу нескольким получателям (здесь: нотификаторам) независимо друг от друга. (Глава 13)
- **Fingerprint (SHA-256)** — криптографический отпечаток содержимого алерта, позволяющий доказать постфактум, что он не был подделан или изменён после генерации. (Главы 1, 7)
- **Гистерезис (в контексте memory pressure)** — разрыв между порогом срабатывания и порогом восстановления, предотвращающий частое переключение состояний при значениях около одной границы. (Глава 15)
- **Graceful shutdown** — упорядоченная остановка агента: сначала прекращается приём новых событий, затем дорабатывается буфер, затем закрываются экспортёры. (Глава 4)
- **Hardware profile (пресет)** — предустановленный набор параметров (`lite`/`balanced`/`production`), подобранный под класс железа и управляющий использованием памяти/CPU. (Главы 4, 22)
- **hostPID / hostNetwork** — режимы пода, при которых он видит PID- и сетевое пространство имён узла, а не изолированное пространство контейнера. (Глава 20)
- **Hot-reload** — автоматическая перезагрузка правил/конфигурации при изменении файлов, реализована через `fsnotify`; для правил обязательна полная валидация нового набора до атомарной замены старого. (Главы 3, 8, 19)
- **IOC (Indicator of Compromise)** — индикатор компрометации: IP, домен или fingerprint, ранее замеченный в связи с атакой. (Глава 15)
- **Index-per-day** — паттерн хранения time-series данных, при котором каждый день (или другой период) данных лежит в отдельном индексе — удобно удалять старые данные целиком. (Глава 14)

## J–P

- **JA3 / JA4** — алгоритмы фингерпринтинга TLS-клиента по форме ClientHello (JA3 — MD5-хэш конкатенации полей, JA4 — структурированный человекочитаемый формат). (Глава 17)
- **JIT (Just-In-Time compilation)** — компиляция проверенного eBPF-байткода в нативные машинные инструкции непосредственно перед выполнением. (Глава 2)
- **Kprobe/kretprobe** — динамическая точка на входе/выходе произвольной функции ядра. (Глава 2)
- **LD_PRELOAD hijack** — техника подмены функций динамически линкуемых библиотек во всех процессах системы через переменную окружения или `/etc/ld.so.preload`. (Глава 15)
- **Lineage (родословная процесса)** — цепочка предков процесса (`parentComm` → `comm` → ...), используемая для детекции характерных цепочек атак. (Глава 9)
- **Линейная память (linear memory)** — единый непрерывный, ограниченный по размеру блок памяти WASM-модуля; вся коммуникация хост↔плагин идёт через запись/чтение по смещениям в этом блоке. (Глава 16)
- **List (Falco)** — именованный список значений, используемый внутри `in (...)`. (Глава 21)
- **Lock ordering (порядок блокировок)** — соглашение о фиксированном порядке захвата нескольких мьютексов, предотвращающее deadlock при пересечении путей кода. (Глава 22)
- **LSM blocklist** — eBPF-мапа (`LsmBlocklist`/`LsmPathBlocklist`), проверяемая прямо в ядре на LSM-хуках, без похода в userspace на каждое событие. (Глава 12)
- **LSM hook (Linux Security Module)** — точка принятия решения о доступе, вызываемая ядром *до* разрешения действия; позволяет не только наблюдать, но и блокировать. (Глава 2)
- **Macro (Falco)** — именованный переиспользуемый фрагмент условия, подставляемый в другие условия по имени. (Глава 21)
- **mapstructure tag** — тег Go-структуры (`mapstructure:"..."`), который viper использует для сопоставления YAML-ключей с полями Go-структуры. (Глава 19)
- **MITRE ATT&CK** — общедоступная база знаний тактик и техник атакующих; используется для маркировки правил и объяснения алертов (`tags`, `ebpf-guard explain`). (Главы 1, 8)
- **mTLS (mutual TLS)** — режим TLS, при котором сертификат предъявляют обе стороны — и клиент, и сервер, а не только сервер. (Глава 13)
- **N-gram (биграмма/триграмма)** — последовательность из N соседних символов; статистика n-грамм по обучающему корпусу позволяет отличить «похожее на язык» от «похожее на случайный набор символов» даже при нормальной общей энтропии. (Глава 17)
- **`opCode`** — целочисленный код оператора условия, вычисленный один раз при загрузке правила для быстрой диспетчеризации через `switch` вместо строкового сравнения на каждое событие. (Глава 7)
- **p99** — 99-й перцентиль латентности: значение, ниже которого укладываются 99% измерений (метрика «типичного худшего случая», в отличие от среднего). (Глава 22)
- **Партиция (partition)** — подмножество Rego-модулей, скомпилированное в отдельный `PreparedEvalQuery` под конкретный тип события — оптимизация скорости оценки. (Глава 10)
- **Persistent flag** — флаг Cobra, наследуемый всеми подкомандами дерева, если не переопределён локально. (Глава 18)
- **PGO (Profile-Guided Optimization)** — механизм компилятора Go, использующий профиль реальной нагрузки (`default.pgo`) для более агрессивной инлайновой оптимизации горячих путей кода. (Глава 22)
- **pidfd** — файловый дескриптор, ссылающийся на конкретный процесс (а не на переиспользуемый PID-номер); устраняет гонку между обнаружением и завершением процесса. (Глава 12)
- **Пост-фильтр (post-filter)** — стадия, которая обрабатывает уже сгенерированные алерты, а не сырые события; Rego-обогащение в ebpf-guard именно такая стадия. (Глава 10)

## Q–Z

- **Rate limiting (per-rule)** — ограничение числа алертов в скользящем окне для одного и того же `RuleID`, не глобально. (Глава 7)
- **RBAC (Role-Based Access Control)** — разграничение прав по ролям; в HTTP API ebpf-guard — как минимум «viewer» (только чтение) и «admin». (Глава 13)
- **Rego** — декларативный язык политик Open Policy Agent, на котором написаны `rules/rego/*.rego`. (Глава 10)
- **Responsible disclosure window** — согласованный период (в ebpf-guard — 90 дней) между приватным сообщением об уязвимости и публичным раскрытием деталей. (Глава 23)
- **Ring buffer** — кольцевой буфер в разделяемой памяти для эффективной передачи событий из ядра в userspace. (Глава 2)
- **Runtime security** — обнаружение и реагирование на атаки **во время выполнения** системы, в отличие от статического анализа образов до запуска. (Глава 1)
- **Schema version (`config_version`)** — версия формата конфигурации, используемая для определения, какие поля устарели/удалены при `config validate`/`config migrate`. (Глава 19)
- **Seccomp-профиль** — список разрешённых syscall'ов на уровне ядра для процесса в контейнере. (Глава 20)
- **`SEC(...)`** — макрос libbpf, определяющий, к какой точке ядра прикрепляется eBPF-программа при загрузке. (Глава 5)
- **ServiceMonitor / PrometheusRule** — Custom Resource'ы Prometheus Operator для авто-обнаружения целей скрейпа и деклараций алертов соответственно. (Глава 20)
- **Shannon entropy** — мера «случайности»/непредсказуемости строки в битах на символ; выше энтропия — ближе к случайному набору символов. Используется и для DNS-DGA детекта (глава 7), и для JA3-контекста (глава 17). (Главы 7, 17)
- **Шард (shard)** — независимый сегмент буфера со своим мьютексом; PID детерминированно определяет свой шард через побитовую маску. (Глава 7)
- **Snapshot** — неизменяемый результат сессии `autolearn`, экспортируемый в YAML-правила и seccomp-профиль. (Глава 11)
- **Socket filter (BPF)** — тип BPF-программы, присоединяемой к сокету для ранней in-kernel фильтрации пакетов до их доставки в userspace. (Глава 17)
- **Strict config mode** — режим (`strict_config: true`), при котором агент отказывается стартовать, если файл конфигурации читаем группой/всеми — защита от утечки секретов через права доступа. (Глава 19)
- **Suppression (подавление)** — запись `(ruleID, comm)` или `comm`, добавленная через feedback API, из-за которой будущие совпадающие алерты отбрасываются до попадания в store/экспортёры. (Глава 11)
- **`struct event`** — общая C/Go структура события, определённая в `bpf/common.h` и зеркально отражённая в `pkg/types/event.go`. (Глава 5)
- **`SyntheticCollector`** — компонент `internal/collector/`, эмулирующий события `pkg/types.Event` для тестов, dry-run и демонстраций. (Глава 3)
- **`sync.Pool`** — механизм переиспользования объектов в Go, используемый коллекторами, чтобы снизить нагрузку на сборщик мусора при высокой частоте событий. (Глава 6)
- **TAP (Test Anything Protocol)** — простой текстовый формат вывода результатов тестов, используемый `ebpf-guard rules check`. (Главы 8, 18)
- **Tracepoint** — стабильная, версионно-независимая точка инструментирования в коде ядра. (Глава 2)
- **Trap** — управляемая аварийная остановка исполнения WASM-модуля (например, деление на ноль или выход за границы памяти), которую хост перехватывает без краха всего процесса. (Глава 16)
- **Upsert** — вставка, которая при конфликте по уникальному ключу превращается в обновление существующей строки (`ON CONFLICT ... DO UPDATE`). (Глава 14)
- **Uprobe/uretprobe** — точка трассировки в **пользовательском** пространстве, привязанная к конкретному адресу/символу в бинарнике или библиотеке (в отличие от kprobe, работающего в ядре); uretprobe срабатывает на возврате из функции. (Главы 2, 17)
- **Verifier** — статический анализатор ядра, проверяющий безопасность eBPF-байткода перед загрузкой (нет неограниченных циклов, нет выхода за границы памяти). (Глава 2)
- **Verifier reject** — отказ верификатора загрузить или присоединить программу, обычно при hot-reload новой версии BPF-кода. (Глава 22)
- **viper** — Go-библиотека для чтения конфигурации из YAML/JSON/env с поддержкой hot-reload через fsnotify. (Глава 19)
- **VPA (VerticalPodAutoscaler)** — контроллер Kubernetes, автоматически подстраивающий CPU/memory requests/limits пода на основе фактического потребления. (Глава 20)
- **WAL (Write-Ahead Logging)** — режим SQLite, в котором изменения сначала пишутся в отдельный журнал, а не в основной файл БД, что позволяет читателям и писателю работать параллельно. (Глава 14)
- **WASI** — стандартный системный интерфейс для WASM вне браузера (файлы, сеть, время); плагины ebpf-guard компилируются под `wasi`, но фактически используют только линейную память, без реального доступа к ОС. (Глава 16)
- **WASM (WebAssembly)** — компактный бинарный байткод-формат, исполняемый в песочнице; здесь используется как безопасный формат для сторонней детект-логики. (Глава 16)
- **XDP (eXpress Data Path)** — точка прикрепления на максимально раннем этапе обработки сетевого пакета, ещё до стека ядра. (Глава 2)
- **Zero-config режим** — запуск агента без файла конфигурации, на встроенных дефолтах и встроенных правилах (`--zero-config`). (Глава 18)
- **Zero-day self-protection (BPF anti-tampering)** — детект попыток вмешаться в уже загруженные BPF-объекты самого агента; не путать с защитой процесса или конфигурации. (Глава 23)
